# Аудит пайплайна и соответствия README (2026-05-07)

## Что проверено
1. Разбиение train/val/test и порядок вычисления статистик.
2. Как данные проходят через dataset -> dataloader -> model -> metrics.
3. Логика обучения, early stopping, выбор лучшей эпохи.
4. Построение ROC/AUC и связанные риски.
5. Насколько narrative-текст README подтверждается кодом.

## Ключевые выводы

### 1) Разбиение и статистики
- В `create_dataloaders` данные сначала делятся на train/val/test через `random_split`.
- Исправлено: нормализация теперь считает mean/std **только по train-индексам после split**, чтобы избежать leakage.

### 2) Архитектура кода (по факту)
- `src/dataset.py`: чтение `.psv`, паддинг/обрезка до `seq_length`, mask по пропускам, target = последний `SepsisLabel`.
- `src/train.py`: обучение по эпохам, BCEWithLogitsLoss, pos_weight из train, warmup+cosine scheduler, early stopping по `val_auc`, сохранение best checkpoint.
- Метрики на валидации: AUROC/F1/ACC; ROC сохраняется через `sklearn.metrics.roc_curve`.

### 3) Проверка «строчка за строчкой» потока данных
- Файл пациента читается в `__getitem__` -> признаки `x`, mask, label `y`.
- Батч из DataLoader передаётся в `model(x, mask)`.
- Лосс считается на logits (`BCEWithLogitsLoss`), вероятности для метрик через `sigmoid`.
- Best epoch выбирается по максимальному `val_auc`.

### 4) ROC/AUC
- Реализация в `train.py` использует стандартные `roc_auc_score` и `roc_curve` из sklearn — корректно технически.
- Но корректность содержательно зависит от источника `y_true/y_prob`.
- В репозитории есть скрипты с **синтетикой/захардкоженными числами** (`scripts/optimize_threshold.py`, `scripts/plot_results.py`), их нельзя считать доказательством реального качества на тесте.

### 5) Несоответствия README реальному коду/процедуре
- Narrative про «предупреждение за 12 часов» не подтверждён явной постановкой задачи в основном `dataset.py` (там target берётся из последнего часа файла).
- Часть скриптов строит графики не из логов обучения, а из заранее заданных массивов.
- Поэтому утверждения в README нужно трактовать как мотивационное описание, а не как строго верифицированный clinical claim.

## Что обязательно сделать для строгой валидации
1. Явно зафиксировать протокол split (seed, список файлов train/val/test) и сохранить его на диск.
2. Считать все статистики/импутацию только из train.
3. Выделить отдельный test-only скрипт, который грузит `*_best.pt`, считает AUROC/F1/PR-AUC/ROC только один раз на test.
4. Строить графики только из артефактов логов (`logs/*.json`, `logs/*roc.csv`), убрать hardcoded-демо как «финальные результаты».
5. Переписать README: разделить «мотивацию» и «строго подтверждённые факты экспериментов».
