# Научрук-ревью курсового проекта (2026-05-07)

## Итоговая оценка
**7/10 (хороший инженерный прототип, но недостаточно строгая научная валидация).**

## Что сделано хорошо
1. **Покрыты несколько архитектур** (LSTM, Transformer, Mamba, GRU-D) и есть единая фабрика моделей.
2. **Есть рабочий training loop** с ранней остановкой, сохранением лучшей эпохи и логированием метрик/ROC.
3. **Учтён дисбаланс классов** через `pos_weight` в BCEWithLogitsLoss.
4. **Есть отдельный трек federated learning** и non-IID сценарий (как исследовательская ветка).
5. **После исправления** нормализация в основном `dataset.py` считается по train после split.

## Критичные недочёты (надо исправить в первую очередь)
1. **Неполная воспроизводимость экспериментов**
   - Нет жёстко сохранённого split-манифеста train/val/test (списки файлов).
   - В отчётах есть цифры, но не везде явно указано, на каком именно split/seed они получены.

2. **Смешение «демо-скриптов» и «научных результатов»**
   - Есть скрипты с синтетикой и захардкоженными метриками/линиями графиков.
   - Это допустимо для демо, но недопустимо как доказательство итогового качества в курсовой.

3. **Оценка на test-set не выделена как финальная неизменяемая процедура**
   - Нужен отдельный `evaluate_test.py` без подбора порога по test.
   - Подбор threshold должен делаться на val, а test использоваться один раз в конце.

4. **Несогласованность артефактов и narrative**
   - Клинические формулировки («за 12 часов предупредит») не подтверждены строго постановкой в коде.

## Средние замечания по коду
1. В `fed_train.py` класс `PhysioNetDataset` загружает только первые 100 файлов клиента (`file_list[:100]`), что может искажать вклад клиентов.
2. В `dataset_timeaware.py` добавлены 42 фичи, но маска остаётся размерности 40; это не ломает текущую модель напрямую, но делает интерфейс неоднозначным.
3. Много скриптов с пересекающейся функциональностью (`train.py`, `train_clean.py`, `train_mamba.py`), нужна консолидация «канонического» pipeline.

## Проверка корректности обучения/данных (по коду)
### Данные
- В `src/dataset.py` чтение `.psv`, обрезка/паддинг до `seq_length`, формирование mask и target из последней строки — логично и консистентно.
- В основном pipeline нормализация теперь train-only (после split), это корректно.

### Обучение
- В `src/train.py` корректно:
  - `BCEWithLogitsLoss` на logits,
  - вероятности через sigmoid,
  - AUROC/F1 на валидации,
  - early stopping и сохранение лучшей эпохи.
- Технически цикл обучения реализован адекватно для baseline-исследования.

### ROC/AUC
- Используются стандартные функции sklearn (`roc_auc_score`, `roc_curve`) — методически корректно.
- Ключевой риск не в формулах, а в источнике `y_true/y_prob` (должны быть только из реального eval-прохода).

## Что доработать для «отлично»
1. Зафиксировать и сохранять split-файлы + seed + версию данных.
2. Сделать единый «production-grade» pipeline:
   - `train.py` (обучение + val)
   - `select_threshold.py` (только val)
   - `evaluate_test.py` (только test, один прогон)
3. Отделить demo-скрипты в `scripts/demo/` и убрать их из секции «итоговые результаты».
4. Добавить минимальные автотесты:
   - shape-check батча,
   - детерминированность split,
   - smoke-тест 1 эпохи на синтетике.
5. Обновить README/отчёт: разделить «мотивацию» и «доказанные экспериментом утверждения».

## Вердикт научрука
Проект уже на уровне **сильного инженерного прототипа** для курсовой. Чтобы это стало **научно аккуратной работой**, нужно жёстко оформить протокол эксперимента и финальную test-оценку, а также убрать неоднозначность между демо и верифицированными результатами.
